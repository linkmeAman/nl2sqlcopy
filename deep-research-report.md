## Where the Current System Stands

Your current system is already past the “toy prototype” stage. Based on the route guide and setup notes you uploaded, you already have a FastAPI service, pgvector-backed retrieval, schema-group and knowledge ingestion, a ReAct-style SQL generation loop, live MySQL column checks, EXPLAIN-based validation, bounded `/ask` execution, learned-pattern storage, interactive instruction teaching, confidence tracking, and streamed progress responses. What is still missing is the machinery that lets the system **generalize** instead of mostly retrieving previously seen guidance. That gap is exactly what newer enterprise-oriented benchmarks highlight. Spider 2.0 was designed around real-world text-to-SQL workflows with very large schemas, long contexts, multiple SQL statements, and dependencies on metadata and documentation; its paper reports that a strong code-agent framework solved only 17.0% of tasks, versus 91.2% on Spider 1.0 and 73.0% on BIRD. ## The Highest-Impact Build Order

1. **Build the evaluation and telemetry harness first.** Freeze an internal benchmark from real traffic before changing the model stack: successful queries, common failures, ambiguous requests, multi-join questions, follow-ups, policy-sensitive queries, and schema-drift regressions. Log, for every request, the routed schema candidates, retrieved chunks, injected rules, SQL candidates, validation failures, EXPLAIN output, execution outcome, latency, and user feedback. Exact string match is not enough; test-suite accuracy and multi-angle evaluation are better release gates for semantic correctness. fileciteturn0file1 citeturn2search5turn2search1turn19search0turn19search8

2. **Replace “vector-only memory” with a typed semantic memory layer.** Keep the vector store, but stop treating it as the primary memory for learned knowledge. User instructions such as term mappings, join rules, metric definitions, default filters, and business rules should be compiled into structured records with scope, provenance, confidence, and schema-version awareness. Your current docs show that instructions and patterns are stored and injected, but some confirmation and embedding flows are still lightweight or manual; Phase 2 should make that durable, typed, and automatically reusable across prompting, retrieval, validation, and testing. fileciteturn0file0 fileciteturn0file1 citeturn3search0turn1search1turn17search0

3. **Add schema routing and hybrid retrieval.** Introduce a dedicated router that first narrows the request to the most likely database, domain, and table family, then runs targeted retrieval at table, column, metric, example-SQL, and business-rule level. Use lexical retrieval for identifiers and exact business terms, dense retrieval for semantic similarity, and graph retrieval for join paths; then fuse ranks, for example with Reciprocal Rank Fusion, before reranking the final context package. This matches the scaling direction in DBCopilot and RASL, which separate schema routing from SQL generation and decompose schemas into retrievable semantic units. citeturn14search0turn14search2turn15search0turn15search3turn5search0

4. **Move from single-candidate generation to plan-generate-select.** Instead of generating one SQL string and validating it afterward, generate a small set of diverse candidates from an explicit intermediate plan, normalize them to an AST, score them against schema/rules, and only then pick the best one for EXPLAIN and bounded execution. DIN-SQL, CHASE-SQL, and execution-guided decoding all point in this direction, and constrained decoding methods such as PICARD materially reduce invalid SQL. citeturn1search0turn4search0turn4search1turn0search2turn0search6

5. **Add proactive ambiguity management and session memory.** Right now your service can return `clarification_needed`, but the route guide shows that this mostly happens after the full ReAct loop runs and gives up. Phase 2 should detect ambiguity earlier, ask one minimal clarifying question, store the answer in session state, rewrite the user query into a canonical form, and continue from there. CoSQL and SParC exist precisely because real NL-to-SQL systems must handle context-dependent, multi-turn questions on unseen databases, and DART-SQL shows that question rewriting plus execution-guided refinement can improve execution accuracy. fileciteturn0file0 citeturn1search2turn1search3turn4search3

6. **Turn user corrections into a continuous improvement pipeline.** A correction should never only boost a pattern count or deactivate a bad example. Every correction should create four assets: a typed error label, a rule or metadata update, a benchmark case, and a training example pair with at least one hard negative. Replay recent failures nightly and retrain lightweight components such as the schema router, reranker, and clarification policy first; only after the dataset is large and clean should you fine-tune the main SQL model. Synthetic-data and self-improving approaches such as SENSE and STaR-SQL support this staged path. fileciteturn0file0 fileciteturn0file1 citeturn13search1turn13search3turn13search0turn13search2

## Building Accuracy and Generalization

### Turn learned instructions into compiled knowledge

The single most important change for generalization is to stop storing user knowledge only as retrievable text and instead compile it into typed, executable knowledge. If a user teaches “counselor means employee table,” that should become a synonym mapping with applicability conditions, not just an embedding. If a user teaches a join or a default exclusion rule, that should update the join graph and validator, not only prompt text. If a user corrects a metric definition, that should become a canonical metric spec with numerator, denominator, grain, permitted dimensions, and default filters. Your current `/teach` flow already distinguishes instruction types such as `table_relationship`, `business_rule`, `term_mapping`, `filter_rule`, and `correction`; Phase 2 should make each of those types first-class, typed artifacts. fileciteturn0file0 fileciteturn0file1

That compiled knowledge should be reused in four places simultaneously: the prompt, the retriever, the validator, and the training/eval pipeline. This is how you move from memorization to abstraction. It is also consistent with recent metadata work: the Automatic Metadata Extraction paper argues that understanding database contents is often the hardest part of query development, and it highlights profiling, query-log analysis, and SQL-to-text generation as useful ways to build the missing metadata layer that many real systems lack. citeturn3search0turn3search8turn0search4

### Add metadata and value understanding

Your system already enriches schema groups with live columns, aliases, and example questions, which is a strong start, but it is still too schema-name centric for the next stage. BIRD was designed specifically to expose the importance of database contents and efficiency, and the metadata-extraction work shows why content-aware metadata matters. Build a metadata profiler that continuously derives column statistics, common values, entity examples, temporal ranges, join-cardinality hints, common filter patterns, and human-readable descriptions from accepted SQL and query logs. Treat this metadata as a separate indexed asset, not just extra text appended to schema chunks. fileciteturn0file1 citeturn10search0turn10search1turn3search0

In practice, create separate stores for table docs, column docs, value profiles, metric definitions, approved joins, and approved query patterns. Retrieve them independently, then compose a concise context package. That component-based retrieval style is close to RASL’s architecture and scales better than pushing larger mixed chunks into one prompt. citeturn15search0turn15search3turn15search10

### Plan first, then generate SQL

Production NL2SQL is now better thought of as a planning problem than a “one prompt in, one SQL out” problem. Spider 2.0 explicitly frames real-world workflows as long-context, sometimes multi-query tasks, and CHASE-SQL shows that multi-path candidate generation plus learned selection outperforms simpler prompting pipelines on BIRD. For your system, that means introducing an intermediate representation that explicitly captures measure, dimensions, time grain, filters, join intent, ranking intent, and output shape before SQL generation starts. citeturn0search4turn4search0turn4search6

Once you have that representation, generate multiple SQL skeletons, fill them with routed schema items, and normalize the result with an SQL parser. SQLGlot is already a practical option if you want a Python-native parser/transpiler/optimizer with broad dialect support, while Apache Calcite is a stronger foundation if you want parser, validator, and rule-based optimizer capabilities. Pair that with constrained decoding such as PICARD so invalid tokens are rejected during generation, not only after generation. citeturn11search0turn11search1turn11search6turn0search2turn0search6

## Ambiguity and Complex Query Handling

### Ask before the system guesses

Your clarification behavior should move from “ask only after the search failed” to “ask whenever a targeted ambiguity detector fires.” The easiest triggers are a low margin between top schema routes, unresolved metric definitions, multiple plausible date columns, multiple valid join paths, disagreement among SQL candidates, or conflicting high-confidence rules. When one of those triggers fires, ask a one-slot clarification such as “Do you mean invoice created date or payment date?” or “By sales, do you mean booked invoice amount or collected payment amount?” Then rewrite the query into a canonical form and continue. That is much closer to the conversational NL2SQL setting reflected by CoSQL and SParC. citeturn1search2turn1search3

Question rewriting is worth making explicit rather than leaving it implicit inside the SQL model. DART-SQL demonstrates that rewriting ambiguous questions using database content and then refining with execution feedback can improve execution accuracy, which is exactly the pattern you want for user-facing clarification loops. citeturn4search3

### Add a small planner for hard queries

Create a planner that classifies requests into a handful of query families: simple lookup, grouped aggregation, top-k ranking, period-over-period comparison, windowed metric, entity drill-down, cohort/funnel, and multi-step workflow. For the easy families, your current guarded path can remain close to what it is. For the harder families, the planner should emit a step sequence such as “resolve metric,” “resolve date grain,” “select join path,” “build aggregation CTE,” “apply ranking,” and “render final projection.” That is broadly aligned with decomposition-centric systems such as DIN-SQL and multi-path systems such as CHASE-SQL, and it is necessary because Spider 2.0 includes richer workflows of exactly this kind. citeturn1search0turn4search0turn0search4

### Explicitly model edge cases and robustness slices

Do not leave common edge cases to prompt luck. Add explicit validators and tests for null semantics, duplicate inflation from one-to-many joins, default soft-delete filters, inconsistent business synonyms, missing unit conversions, and date/time ambiguity. Dr.Spider is useful here because it was built as a robustness benchmark with 17 perturbations and showed that even robust systems still suffer sizable drops under realistic perturbations. Your own benchmark suite should include analogous perturbation slices for your schema and terminology. citeturn2search0turn2search4

## Evaluation and Feedback Loops

A production-minded eval stack for your system should operate at four layers. First, measure retrieval quality: table recall@k, column recall@k, join-path recall, and reranker NDCG. Second, measure planning quality: intent or IR slot accuracy, ambiguity detection precision and recall, and clarification success rate. Third, measure SQL quality: syntax-valid rate, validator pass rate, execution accuracy, and test-suite accuracy. Fourth, measure product quality: first-pass success rate, clarification rate, correction rate, p50/p95 latency, token cost, and safe-abstain rate. The reason to be this explicit is that exact-match evaluation misses semantic equivalence, while frameworks such as NL2SQL360 were built for finer-grained comparisons across domains and SQL characteristics rather than one aggregate score. citeturn2search5turn2search1turn19search0turn19search12

Build your internal benchmark set from real traffic, but slice it deliberately. At minimum, maintain slices for single-table queries, standard joins, multi-hop joins, aggregations, date intelligence, follow-up questions, ambiguous questions, policy-sensitive queries, and schema-drift regressions. Add Dr.Spider-style perturbation slices and Spider 2.0/BIRD-style large-context slices so improvements do not silently overfit only the easy path. citeturn2search0turn0search4turn10search0

Each correction should flow through a durable pipeline. The first stage labels the error source: routing, schema linking, metric semantics, filter semantics, value grounding, SQL composition, validator over-blocking, or answer summarization. The second stage updates the right asset: rule store, metadata store, join graph, retriever labels, or training corpus. The third stage creates or updates a benchmark case. The fourth stage schedules replay and, when enough similar cases accumulate, distills them into additional training data for the router, reranker, or generator. That is a much richer loop than your current pattern boost/deactivate path and confidence decay, and it is the right precondition for later self-improving training. fileciteturn0file0 fileciteturn0file1 citeturn13search1turn13search0

## Scalable Architecture and Retrieval

Your current service can remain the orchestrator, but Phase 2 should separate the online path from the offline learning path. The online path should be latency-sensitive and deterministic: gateway, session state, router, retrievers, planner, candidate generator, validator, executor, answerer. The offline path should be durability- and throughput-oriented: schema ingestion, metadata profiling, query-log mining, synthetic-data generation, benchmark building, retraining, replay evaluation, and rule conflict review. This split matters because some of your current behaviors are intentionally lightweight or ephemeral — for example, fire-and-forget background tasks and in-memory confirmation tokens — and those are not strong enough for production durability. fileciteturn0file0 fileciteturn0file1

For storage, keep a relational source of truth for typed metadata and rules, and use retrieval systems as specialized indices over that truth. A practical layout is: relational tables for schemas, columns, joins, metrics, rules, feedback events, and benchmark cases; vector indices for table descriptions, column descriptions, examples, paraphrases, and approved query patterns; a lexical index for exact identifier matches and abbreviations; and an adjacency graph or relation table for join-path search. That structure mirrors the schema-routing and component-retrieval direction of DBCopilot and RASL. citeturn14search0turn14search2turn15search0turn15search3

On the pgvector side, you still have room to get faster and more reliable without leaving Postgres. The official pgvector docs note that exact nearest-neighbor search gives perfect recall, while approximate indexes trade recall for speed. HNSW offers a better speed–recall tradeoff than IVFFlat but uses more memory and builds more slowly; `halfvec` can reduce index size; filtered approximate search can under-return results unless you tune `hnsw.ef_search` or enable iterative scans; and partial indexes or partitioning become attractive when filtering patterns are predictable. In Phase 2, partition embeddings by tenant, artifact type, and schema version, add exact B-tree filters for common metadata predicates, and benchmark when to fall back to exact search for small candidate sets. citeturn18view1turn18view0turn18view2

For guardrails, keep your current read-only checks, row caps, and EXPLAIN usage, but harden them. MySQL’s EXPLAIN is designed to reveal how the database will execute a statement, including join order, and MySQL supports per-statement execution ceilings through `MAX_EXECUTION_TIME(N)`. Combine that with a read-only database user, SQL normalization through SQLGlot or Calcite, bounded result size, audit logging, and a sandbox executor that can reject high-cost plans before they hit production data. Your uploaded docs already recommend read-only production use and bounded execution; Phase 2 should make those guarantees stricter and more observable. fileciteturn0file0 fileciteturn0file1 citeturn7search0turn7search11turn16search0turn11search0turn11search1

## The Production-Grade Target

A real “Phase 2” version of your system should look like this:

```text
User query
  -> intent and ambiguity detector
  -> schema router
  -> hybrid retrievers
  -> planner and intermediate representation
  -> multi-candidate SQL generation
  -> parser, validator, cost and policy checks
  -> bounded execution sandbox
  -> answer composer
  -> feedback capture
  -> replay, retraining, and benchmark update pipeline
```

Operationally, that system should have three trustworthy outcomes: answer directly when confidence is high and checks pass, ask a precise clarification when the uncertainty is narrow and resolvable, or abstain safely when the request is unsupported or risky. It should version all learned artifacts against schema revisions, persist all feedback and review state, make it easy to tell whether a failure came from routing, retrieval, planning, generation, or execution, and gate every release on a standing benchmark plus recent-traffic replay. That is the production standard implied both by the limits shown in enterprise-oriented benchmarks and by the specific durability gaps that still exist in your current implementation. fileciteturn0file0 fileciteturn0file1 citeturn0search4turn10search0turn19search0
